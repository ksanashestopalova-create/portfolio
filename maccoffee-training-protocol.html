Operator Onboarding Protocol

Workplace Researcher · MacCoffee · Stick-packaging line · 2014–2016
People involved in the protocol
Maintenance engineers
Equipment setup, safety rules, supervised first shift
Quality department head
Raw material testing, instruction verification, shift sign-off
Safety engineer
Safety rules, instruction verification, independent shift check
Researcher (me)
Instruction drafting, operator interview, live documentation, iteration
Onboarding phases
Phase 0 Equipment readiness — before any operator touches the machine Maintenance engineers + Safety engineer + Production manager
Engineers study machine documentation independently
Machine connected and initial test run completed — "what does this thing actually do"
Engineers and safety specialist jointly define baseline safety rules: where not to put hands, when to shut down, emergency procedure
Three test runs with different real coffee blends (prepared by food technologists) — coarser and finer grinds seal differently and require different machine settings
Quality department head personally reviews all test output, selects best samples, identifies machine settings that produced them
Optimal settings data passed to researcher for instruction drafting
↓
Phase 1 Instruction creation and verification Researcher · Quality head · Safety engineer
Researcher drafts instruction from test run data and engineer observations
Draft verified by quality department head and safety engineer independently
First operator introduced to the draft — reads, then is orally assessed by researcher
Practical test: operator demonstrates procedures at the machine, assessed by maintenance engineer
The draft is intentionally a draft — it is expected to be incomplete. First operator interaction will surface what is obvious to engineers but non-obvious to operators.
↓
Phase 2 First supervised shift — instruction iteration in real time Operator + Maintenance engineer (supervisor) + Researcher (observer)
Operator runs machine for full 12-hour shift under maintenance engineer supervision
Researcher observes and documents gaps between written instruction and real operation — film tension, timing cues, tacit knowledge held by engineers
Post-shift: researcher and operator review instruction together, adding everything that was non-obvious in practice
Instruction updated, verified, and approved as final version
The operator is a co-author of the final instruction. They have knowledge engineers don't have — what it feels like to learn this for the first time.
↓
Phase 3 Independent operation with competency verification Operator (independent) + Quality specialist + Safety specialist
Operator runs second shift independently
Quality specialist checks output against standard — independent of operator
Safety specialist observes protocol adherence — independent of operator
If both sign off: operator is certified for independent operation on this line
↓
Phase 4 Subsequent operators — peer-led onboarding New operator + Certified operator (peer supervisor)
New operator receives final approved instruction and simplified reference material from researcher
First shift: works under direct supervision of certified operator — peer-to-peer knowledge transfer
From second shift: works independently
Why the operator is a co-author — not just a trainee

The instruction written before operator involvement was necessarily incomplete. Engineers know how the machine works mechanically. They do not know what is non-obvious to someone learning it for the first time — which controls feel ambiguous, which physical cues (film tension, sound, resistance) are easy to miss, where anxiety is highest. The first operator's experience of learning produced knowledge that no engineer could have generated. The final instruction reflects both sources.

Built-in feedback loop

The protocol was designed to improve itself. Phase 2 is not just training — it is structured observation that feeds directly back into the instruction. Each new equipment type onboarded through this protocol produces a better instruction than could have been written from documentation alone. The system learns as it runs.
